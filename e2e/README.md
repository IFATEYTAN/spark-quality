# בדיקות E2E (Playwright)

בדיקות מקצה-לקצה ל-SPARK Quality. מתמקדות ב**דמו הציבורי** (`/demo`) ובדפי השיווק —
אזורים שאינם דורשים התחברות או מסד נתונים, כך שהן רצות מול `pnpm dev` כמו שהוא.

## הרצה

```bash
# פעם אחת — התקנת דפדפן (דורש גישת רשת ל-cdn.playwright.dev)
pnpm test:e2e:install

# הרצת כל הבדיקות (Playwright מריץ את שרת ה-dev אוטומטית)
pnpm test:e2e
```

> ⚠️ בחלק מהסביבות (כולל ה-sandbox של Claude) הורדת הדפדפן חסומה ע"י מדיניות
> רשת. הריצו את הבדיקות היכן ש-`cdn.playwright.dev` נגיש (CI / סביבת Manus /
> מחשב מקומי).

## מה מכוסה
- **`marketing.spec.ts`** — דף הבית, תמחור (₪199), תקנון נטענים.
- **`demo.spec.ts`** — ניווט לשלב גריד 16 הטריגרים, אימות שכל 15 התוויות מוצגות,
  ולחיצה על טריגר פותחת את מודאל התרחיש הנכון עם התוכן המתאים.

## מה עדיין לא מכוסה — האזור המחובר
מסכי המוצר (`/dashboard`, `/clients`, `/tasks`, `/analytics`, `/team`, `/admin`)
דורשים **התחברות (Manus OAuth) + מסד נתונים עם נתוני-זרע**. כדי לבדוק אותם מקצה-
לקצה צריך תחילה:
1. מנגנון **test-auth** (עוקף OAuth) או session cookie לבדיקות.
2. **DB בדיקות** עם workspace + לקוחות + פוליסות זרועים (כדי לאמת שכל טריגר מציג
   את הנתונים הנכונים).
3. ניקוי נתונים אחרי כל ריצה.

זו תשתית שדורשת החלטה על סביבת הבדיקות (מפתחות/DB) — ראו את הדיון מול הצוות.
