> מסמך זה מסכם את מצב המוצר נכון ל-16 ביוני 2026: מה מוכן ועובד מקצה לקצה, מה
> בשלב שיפור, ומה דורש סגירה רגולטורית/משפטית. מיועד לשיתוף פנימי (ענת) ולמשקיעים.

---

## 1. תקציר מנהלים (Bottom Line)

**המערכת עובדת מקצה לקצה כמוצר מסחרי — כולל גביית תשלום מלקוחות.**

הלולאה העסקית המלאה סגורה ופרוסה במערכת החיה:

> הרשמה → הגדרת סוכנות → **תשלום (iCount דרך Make)** → הפעלת מנוי → העלאת דוח
> surense → סיווג אוטומטי ל-16 טריגרים → דשבורד ורשימת פעולות → ניסוח הודעות ב-AI
> → משימות ותזכורות → מחיקת מידע ומדיניות שמירה.

ניתן להתחיל לקבל לקוחות משלמים. הנושאים הפתוחים הם שיפורים (שליחת WhatsApp
אוטומטית) ועטיפה רגולטורית — לא חסמים לתפעול הבסיסי.

## 2. מבנה ארכיטקטורי

| שכבה | תפקיד | טכנולוגיה |
|---|---|---|
| **ממשק (Frontend)** | מסכי הסוכן, עיצוב "Editorial Fintech" RTL | React + Tailwind |
| **שרת ולוגיקה (Backend)** | פענוח דוח, סיווג ל-16 טריגרים, ניסוח AI, הרשאות | Express + tRPC |
| **מסד נתונים** | אחסון מבודד לכל סוכנות | MySQL / TiDB |
| **AI** | ניתוח וניסוח הודעות | Claude Sonnet 4.6 |
| **חיוב** | סליקה וחיוב | iCount דרך Make |

**עקרון יסוד:** בידוד רב-דיירי (multi-tenant) — כל סוכנות רואה אך ורק את הנתונים
שלה; הקובץ הגולמי שמועלה אינו נשמר בשרת, רק הנתונים המחולצים.

## 6. לסגירה לפני קנה מידה (רגולטורי/משפטי) 🔴

אלה אינם קוד — דורשים יועץ משפטי / תשתית:

- **הצפנת מידע במנוחה** (encryption at rest).
- **הרשמת מאגר מידע** ברשם מאגרי המידע.
- **הסכם עיבוד נתונים (DPA)** מול הסוכנים, והסכמת לקוח קצה.

> חשוב במיוחד כי המערכת מטפלת ב-PII רגיש (ת"ז, נתונים פיננסיים) תחת חוק הגנת
> הפרטיות (כולל תיקון 13).
